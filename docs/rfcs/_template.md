## Summary

One paragraph. What is being proposed? Read this and stop, and the reader should understand the shape of the change.

## Motivation

What problem does this solve? What is wrong with the current state, or what opportunity is being missed?

## Detailed proposal

The substantive content of the proposal. Long enough to be specific; short enough to be readable in one sitting. Use sub-sections if needed.

If the proposal affects specific files in the repository, list them. If it implies a new ADR, say so.

## Alternatives considered

What other approaches could solve the same problem? Why are they not being proposed?

## Implementation notes

If accepted, what does implementation look like? Who does the work? How is it phased? What dependencies exist?

## Open questions

What is genuinely unclear or contested in this proposal? List them explicitly. The RFC process is a chance to resolve open questions before implementation begins.

## Notes for authors

- RFCs are for **proposals**, not for **decisions**. They are how thinking is shared before it becomes binding.
- A successful RFC leads to either an implementation PR (referencing the RFC) or one or more new ADRs (capturing decisions the RFC resolved).
- RFCs that are rejected stay in the repository — they document what was considered and why it was not adopted. This is valuable history.
- Keep RFCs focused. If a proposal touches three orthogonal things, write three RFCs.
- The target decision date keeps RFCs from rotting. If an RFC hasn't been resolved by the target date, decide explicitly: extend, accept, reject, or withdraw.
